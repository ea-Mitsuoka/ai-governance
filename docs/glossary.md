## 3. 基本AI用語(AI事業者GLに準拠)

| 用語 | 定義 |
|---|---|
| **ハルシネーション** | AIが事実と異なる内容をもっともらしく生成する現象。出力検証を必須とする前提(P7) |
| **グラウンディング** | AIの出力を信頼できる情報源(社内文書等)に紐づけ、根拠を持たせること |
| **RAG** | Retrieval-Augmented Generation(検索拡張生成)。AIが外部・社内のドキュメントを検索し、その内容を根拠に回答を生成する仕組み。社内Geminiの基本構成 |
| **エージェント / エージェンティックAI** | 自律的に手順を決め、ツールを実行し、複数ステップで目標を達成するAI。単なる応答生成より自律度が高く、リスクも質的に異なる |
| **HITL** | Human-in-the-Loop。AIの処理の途中に人間の承認・確認を挟む設計。不可逆操作・重要判断で必須([04](to-be/04-operational-flows.md)) |
| **A2A** | Agent-to-Agent。エージェント間の通信。マルチエージェント環境での監査対象に含む |

## 4. リスク・脅威フレームワーク(外部標準)

| 用語 | 定義 |
|---|---|
| **ASI01〜10** | OWASP「Top 10 for Agentic Applications 2026」のエージェント上位10リスク。リスク台帳の**一次分類**([要約](../sources/2025-12_owasp_top10-for-agentic-applications-2026.md)) |
| **T1〜T17** | OWASP「Agentic AI – Threats and Mitigations」の詳細脅威タクソノミ。ASIの**細目(二層目)**([要約](../sources/2025-12_owasp_agentic-ai-threats-and-mitigations_v1.1.md)) |
| **PB1〜PB6** | 同文書の6つの対策プレイブック。各々Proactive/Reactive/Detectiveの3層構成 |
| **二層構造** | リスク台帳の分類方式。ASI01-10(一次)× T1-17(細目)で脅威を位置づけ、対策(PB)まで縦串で引ける([01](to-be/01-risk-classification-and-grading.md)) |
| **Least-Agency** | 最小エージェンシー。最小権限(Least-Privilege)を拡張し、権限だけでなく「自律性そのもの」を必要最小限に絞る原則(P5/設計判断3) |
| **AIVSS** | AI Vulnerability Scoring System。AIリスクの定量スコア化(将来採用候補) |
| **広島AIプロセス国際行動規範** | AI開発者・提供者向けの国際的な行動規範。e-Agencyが提供者・開発者の立場をとる際の基準 |
| **EU AI Act** | EUのAI規制。リスク階層(許容不可/高/限定/最小)を格付け基準の参考にした |

## 5. 統制・運用・技術用語

| 用語 | 定義 |
|---|---|
| **ゼロトラスト** | 「自然言語入力はすべて非信頼として扱う」前提。社内ユーザも攻撃者になりうるとして性善説で組まない(P3/OWASP ASI) |
| **Secure by Design** | 後付けでなく設計段階から統制を組み込む考え方(P6) |
| **observability(可観測性)** | 誰が・何を入力し・何が返り・何を実行したかを観測・追跡できる状態。エージェント時代は「非交渉(必須)」 |
| **キルスイッチ** | 暴走・侵害時にエージェントを即時停止する仕組み。インシデント初動の封じ込め手段 |
| **クロステナント・ベクトル漏出** | RAGで、本来アクセス権のない他テナント(他部署等)の機密chunkが類似度検索経由で漏れること。名前空間フィルタで防ぐ |
| **bootstrap poisoning** | AI自身の出力を無検証で信頼メモリに再取込みし、誤りが自己強化される汚染。再取込み禁止で防ぐ |
| **CASB** | Cloud Access Security Broker。未承認クラウド・AIサービスの利用を検知する仕組み。シャドーAI対策で利用 |
| **シャドーAI** | 会社が承認していないAIサービスを業務に使うこと。可視化・抑止の対象 |
| **SBOM / AIBOM** | Software / AI Bill of Materials。ソフト・AIの構成部品一覧。サプライチェーン評価で確認 |
| **DLP API / VPC-SC / KMS** | S4技術実装で想定するGoogle Cloudの統制基盤(データ漏えい防止/境界制御/鍵管理) |
| **J-SOX** | 金融商品取引法に基づく内部統制報告制度。プロンプト・メモリ定義のバージョン管理をこの統制の拡張として位置づける |
| **MAESTRO / STRIDE** | 脅威モデリング手法。MAESTRO=層別、STRIDE=脅威分類の古典。OWASP T&Mが参照 |

## 6. 法務用語

| 用語 | 定義 |
|---|---|
| **依拠性** | 著作権侵害成立要件の一つ。既存著作物に「依拠して」作成したか。AIへの既存著作物の入力が論点(文化庁2段階モデル) |
| **類似性** | 同じく侵害要件。生成物が既存著作物と表現上似ているか。出力段階で人間レビューで確認 |
| **営業秘密 / 秘密管理性** | 不正競争防止法の保護対象。秘密として管理されている(秘密管理性)等の要件を満たす情報。セキュリティ対策はこの要件充足を基礎づける有利な一要素になる |
| **規範的行為主体責任** | AIの出力で権利侵害等が起きた場合の事業者の責任。出力フィルタ等の抑止措置で軽減できる |
| **別添7A/B** | 総務省と経済産業省が共同で策定したAI事業者ガイドライン附属のチェックリスト。7A=利用者向け、7B=開発者・提供者向け。「何をすべきか」の網羅的確認リスト |
| **別添7C** | 別添7A/Bの次段にあたる「具体的なアプローチ検討のためのワークシート」。7A/Bで確認した項目について、1ユースケース1枚でリスクアセスメントし、具体的な対応方針を検討する様式。承認制案件の評価・外販テンプレの骨格に使う([ワークシート](../sources/2026-03-31_meti-soumu_ai-business-guideline_v1.2_worksheet.md)) |
