# CLAUDE.md — ai-governance プロジェクト運用ルール

このリポジトリは、e-Agency の AIタスクフォース「内部統制」分科会(AIリスク・内部統制・セキュリティ担当)の作業場。Gemini Enterprise 導入を自社へ適用(クライアントゼロ)し、そのノウハウを外販パッケージ化する親PJの一部。

## このプロジェクトの全体方針

- 進め方は [roadmap.md](roadmap.md) の5ステージ(S0土台 → S1外部基準 → S2理想形 → S3自社ルール → S4技術実装 → 改訂ループ)に従う。
- 成果物は常に **社内適用版** と **外販テンプレート版** の二形態を意識する。
- 親PJフェーズ2「ガバナンス策定」(2026/7納期)に向け、起点3点(機密データフィルタリング/プロンプト監査ログ監視/AI生成物の著作権・法的適合性)を優先縦断する。

## ファイル命名規則

- ファイル名は **小文字の英単語**(語の区切りはハイフン `-`、日付前置は `YYYY-MM-DD`)。例: `roadmap.md`, `2026-04-30_ai-taskforce-charter_draft.pdf`。
- 例外(特別な慣習名)は元のまま: `CLAUDE.md`。
- 日本語のファイル名は使わない(原典が日本語名でも保管時に英語小文字へリネーム)。

## ナレッジ蓄積ルール

- 外部基準・ガイドラインはユーザーが1件ずつ添付する。読んだら `sources/` に「要約MD + 原典」をペアで保管。
- 各要約には必ず **出典URL・該当ページ・自社への適用示唆・起点3点との関連** を含める。
- `sources/index.md` に索引を追記する。

## ★継続的な依頼:ジャーナル(開発日記)の記録

ユーザーは、ガバナンス策定 → 体制構築 → 技術対応 → 完成 に至る全過程を、最終的に**ブログ記事として発表**する予定。そのための一次記録として [journal.md](journal.md) に日記形式で継続的に書き残すこと。

- **書くタイミング**: 各セッションで意味のある進捗・意思決定・気づき・つまずきがあったとき(節目ごと。些末な操作は不要)。
- **書く内容**: その日に何を・なぜやったか、検討した選択肢と採否の理由、判明した事実、次の一手。結果に加え判断の理由も簡潔に残す(後でブログにする際の素材になる)。
- **文体**: 一人称・簡潔・淡々。感情過多にしない一方、完全に無機質にもしない(事実+判断の理由+短いメモ程度の温度)。日付見出し + 本文 + 末尾に「- メモ」or「- 次の一手」。
- **新規エントリは追記**(過去の記述は改変しない。事実誤りの訂正は追記の形で)。
- 日付は会話コンテキストの currentDate を使う(`Date` 関数は使わない)。

## メモリ

- 長期メモリは `~/.claude/projects/.../memory/`(別管理)。プロジェクトの恒久的事実はそちらにも。
