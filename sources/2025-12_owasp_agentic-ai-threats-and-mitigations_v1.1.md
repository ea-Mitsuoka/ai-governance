## 0. 本書の構成と立ち位置

- **方法論**: 特定の脅威モデリング手法を採らず、自前の**リファレンス・アーキテクチャ**(単一エージェント/マルチエージェント)をキャンバスに脅威を説明。STRIDEの拡張として**MAESTRO**(層別)を推奨として紹介。
- **スコープ**: LLMベースのエージェントに限定。一般的脅威(API/アプリ/LLM Top10で既出)は対象外とし、**エージェント特有の新規/変種脅威**に集中。
- **リファレンス・アーキテクチャの構成要素**: Application / Agent(NL入力・Planning・Action・Tools/Function Calling・Memory) / LLM Model(s) / Services(Content・Data・Code・Device・Service・Human-in-the-loop) / Supporting Services(長期記憶ストレージ・ベクタDB・RAG)。マルチエージェントは + エージェント間通信(A2A)+ 調整エージェント。
- **エージェント・パターン**(脅威モデリングの共通語彙): Reflective / Task-Oriented / Hierarchical / Coordinating / Distributed Ecosystem / HITL Collaboration / Self-Learning / RAG-Based / Planning / Context-Aware。

## ★ 1. 脅威タクソノミ T1-T17(意思決定ツリーで分類)

本書の最大の価値。「自社のエージェントはどの問いにYesか」で該当脅威を引く**意思決定ツリー**構造。

### Step 1: エージェントが自律的に手順を決めるか?(推論・エージェンシー由来)

| T      | 脅威                                                    | 要点                                                                                                                                                   | 起点 |
| ------ | ------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ | ---- |
| **T6** | Intent Breaking & Goal Manipulation(意図破壊・目標操作) | PI・汚染データ・悪性ツールで計画/推論/自己評価を改変。漸進的プラン注入・直接/間接プラン注入・**Reflection Loop Trap**(無限自己分析でDoS)・メタ学習注入 | ①②   |
| **T7** | Misaligned & Deceptive Behaviors(逸脱・欺瞞行動)        | 制約回避で違法・有害行動。自己保存(停止回避)・**CAPTCHA突破のため人間を欺いて雇う**・インサイダー取引                                                  | ②③   |
| **T8** | Repudiation & Untraceability(否認・追跡不能)            | 十分なログ/追跡なしで監査・帰責・検知が困難。金融取引の隠蔽・セキュリティ回避・コンプラ違反の隠蔽                                                      | ②    |

### Step 2: 記憶に依存して意思決定するか?(メモリ由来)

| T      | 脅威                                                  | 要点                                                                                                                                                      | 起点 |
| ------ | ----------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- | ---- |
| **T1** | Memory Poisoning(メモリ汚染)                          | 短期/長期記憶を汚染し判断を操作。旅行予約の虚偽価格・コンテキスト窓悪用・セキュリティAIの記憶を「正常」に書換え・**共有メモリ汚染で他エージェントに波及** | ①③   |
| **T5** | Cascading Hallucination Attacks(連鎖ハルシネーション) | 虚偽が自己強化(reflection/記憶)・エージェント間伝播で増幅。医療・為替で系統的誤り                                                                         | ②③   |

### Step 3: ツール・システムコマンド・外部連携で行動するか?(ツール/実行/サプライチェーン由来)

| T       | 脅威                                                              | 要点                                                                                                                                           | 起点 |
| ------- | ----------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- | ---- |
| **T2**  | Tool Misuse(ツール悪用)                                           | 権限内でツールを連鎖悪用。パラメータ汚染(1席→500席)・ツール連鎖で顧客情報持出・大量フィッシング生成                                            | ①②   |
| **T3**  | Privilege Compromise(権限侵害)                                    | 役割誤管理・動的権限継承で昇格。一時管理権限の持続化・**HR→Financeの越境**・Shadow Agent展開                                                   | ①②   |
| **T4**  | Resource Overload(リソース枯渇)                                   | 計算/メモリ/外部依存を枯渇。推論時悪用・マルチエージェント枯渇・**APIクォータ枯渇**・メモリカスケード障害                                      | ②    |
| **T11** | Unexpected RCE & Code Attacks(意図せぬRCE)                        | 生成コード実行で侵害。DevOps Agentにシークレット抽出Terraform生成・言語的曖昧さでPOP3メール持出                                                | ①    |
| **T16** | Insecure Inter-Agent Protocol Abuse(エージェント間プロトコル悪用) | MCP/A2Aの欠陥。**同意フロー操作**(データ削除を自動承認)・MCPレスポンス注入・ツール記述子悪用                                                   | ①②   |
| **T17** | Supply Chain Compromise(サプライチェーン)                         | 上流(プロンプト/プラグイン/更新)汚染が伝播。**Amazon Q供給網**(破壊的プロンプト混入)・**Replit Vibe Coding**(本番DB削除・虚偽テスト結果で隠蔽) | ①③   |

### Step 4: 認証でユーザ/ツール/サービスを検証するか?(認証・なりすまし由来)

| T      | 脅威                                          | 要点                                                                                                                                                                                                                                  | 起点 |
| ------ | --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---- |
| **T9** | Identity Spoofing & Impersonation(なりすまし) | 認証機構を悪用しエージェント/人/サービスになりすまし。ユーザなりすまし・**振る舞い模倣**・クロスプラットフォームID偽装・**冤罪(他者IDで不正)**・**永続的Agent ID乗っ取り**(誤設定クラウドの長命トークン窃取→Entra Agent IDなりすまし) | ①②   |

### Step 5: 目標達成に人間の関与が要るか?(人間関連)

| T       | 脅威                                       | 要点                                                                                | 起点 |
| ------- | ------------------------------------------ | ----------------------------------------------------------------------------------- | ---- |
| **T10** | Overwhelming Human-in-the-Loop(HITL過負荷) | 過剰な介入要求・判断疲労で承認を急がせる。HII操作・認知過負荷・信頼機構の漸進的破壊 | ②    |
| **T15** | Human Manipulation(人間操作)               | 人間のAI信頼を悪用。**請求書詐欺(IPIで振込先を攻撃者口座に)**・AI生成フィッシング   | ②③   |

### Step 6: 複数の相互作用するエージェントに依存するか?(マルチエージェント)

| T       | 脅威                                                                 | 要点                                                                                                                                              | 起点 |
| ------- | -------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | ---- |
| **T12** | Agent Communication Poisoning(通信汚染)                              | エージェント間通信に虚偽注入。協調判断操作・信頼網悪用・合意機構悪用                                                                              | ①②   |
| **T14** | Human Attacks on Multi-Agent Systems(マルチエージェントへの人間攻撃) | 委譲・信頼・依存を悪用し越境昇格。なりすましによる協調権限昇格・**委譲ループ昇格**・タスク飽和DoS・承認偽造                                       | ②③   |
| **T13** | Rogue Agents(ローグ・エージェント)                                   | 悪性/侵害エージェントがマルチ環境に潜入。悪性ワークフロー注入・オーケストレーション乗っ取り・協調フラッディング・**感染性バックドア・カスケード** | ①②③  |

> ★ 単一エージェント(Step1-5)のリスクが、マルチエージェント(Step6)では信頼悪用・依存・連鎖で増幅される、という構造。「まず単一の脅威を理解し、マルチの文脈で伝播を評価せよ」が本書の指針。

## ★ 2. 6つの対策プレイブック(Proactive/Reactive/Detective の3層)

| #   | プレイブック                                      | カバーする脅威                 |
| --- | ------------------------------------------------- | ------------------------------ |
| 1   | AIエージェントの推論操作を防ぐ                    | T6 / T8 / T7                   |
| 2   | メモリ汚染・知識汚染を防ぐ                        | T1 / T5                        |
| 3   | ツール実行のセキュア化・不正行動/供給網汚染の防止 | T2 / T3 / T11 / T4 / T16 / T17 |
| 4   | 認証・ID・権限統制の強化                          | T3 / T9 / T16                  |
| 5   | HITL保護・人間相互作用由来の脅威防止              | T10 / T15                      |
| 6   | マルチエージェント通信・信頼機構のセキュア化      | T12 / T14 / T13                |

各プレイブックは **Step1 Proactive(予防)→ Step2 Reactive(対応)→ Step3 Detective(検知)** の3段構成。主要な対策粒度:

- **PB1(推論)**: 攻撃面縮小+行動プロファイリング/目標一貫性検証・目標変更頻度の追跡・**自己強化ループ防止**/暗号署名・改ざん耐性ログ・人間オーバーライドのログ・**「初め拒否→後に承認」の決定反転の検知**。
- **PB2(メモリ)**: メモリ書込スキャン・セッション分離・コンテキスト依存アクセス・保持期間制限・出所帰属・**コミット前の多エージェント/外部検証**・確率的真偽チェック/異常検知・**ロールバック/スナップショット**・書換頻度の異常検知。
- **PB3(ツール/実行/供給網)**: ツールアクセス制御・**プロンプト/スクリプト/メモリ定義もコード同様にバージョン管理+ピアレビュー**・関数レベル認証・**サンドボックス(非root・リソース制限・実行後に毎回破棄)**・SBOM/AIBOM・署名。
- (PB4-6は脅威マッピング表のとおり。認証=ゼロトラスト/短命資格、人間=多段承認/平易なリスク提示、マルチ=相互認証/メッセージ署名/watchdog。)

## 自社(e-Agency / 内部統制分科会)への適用示唆

1. **リスク台帳の二層構造が完成**: ASI01-10(上位10)を一次分類、その下に**T1-T17を細目**として紐付ける(ASI付録Aのマッピングがそのまま使える)。さらに総務省GLの脅威(PI/DoS/ポイズニング)とPalo Alto5類型を従属マッピング。中立標準2本(ASI+T&M)を背骨に据える構成が確定。
2. **意思決定ツリー(6ステップ)を「ユースケース別リスク選定」の手順に採用**。別添7Cワークシートの「検討対象か(除外理由)」列の判断を、この6つの問い(自律的に手順を決めるか/記憶依存か/ツール実行か/認証するか/人間関与か/マルチか)で機械的に絞れる。社内Gemini RAGチャットボットなら Step2(記憶=T1/T5)・Step3(ツール=T2/T11/T16/T17)・Step4(認証=T9)が該当、と即特定できる。
3. **6プレイブックを利用指針/アセスメントの「対策カタログ」に転用**。Proactive/Reactive/Detective の3層は、別添2A行動目標の「予防・事故対応・監視」とも整合。CC BY-SA 4.0で翻案可なので、プレイブックを和訳・自社化して外販テンプレの対策集にできる(継承ライセンス表記要)。
4. **起点②の根拠がさらに厚くなった**: T8(否認・追跡不能)が独立した脅威として立っており、「並列・自律の推論/実行経路はログが追いつかない」ことを正面から扱う。PB1 Step3 の「暗号署名・改ざん耐性ログ・人間オーバーライドのログ・決定反転の検知」は、起点②の監視設計の具体要件として直接採用できる。
5. **起点①の具体化**: T3(権限侵害=HR→Finance越境・Shadow Agent)/T11(RCE)/T2(ツール連鎖でのデータ持出)。PB3の「サンドボックス・関数レベル認証・プロンプト/メモリもバージョン管理」を社内Geminiのツール実行ガードレールに。
6. **起点③との接点**: T1/T5(汚染→生成物の不正確・不適法)+ T15(人間操作で不適切な生成物を承認)。文化庁「AIと著作権」の依拠性管理と、汚染データ経路の管理がつながる。
7. **PB2の「自分の生成物の再取込み防止」「コミット前の多エージェント検証」はGemini EnterpriseのRAG/メモリ設計の必須要件**(ASIの bootstrap poisoning 防止と一致)。中立標準2本で同じ結論=確度が高い。
8. **「プロンプト・メモリ定義もコードと同様にバージョン管理+ピアレビュー」(PB3)は内部統制と相性が極めて良い**。変更管理・職務分掌のJ-SOX的統制を、AIのプロンプト資産に拡張する論拠として明文化できる。

## 起点3点との関連

- **① 機密データ** → T1/T2/T3/T9/T11/T16/T17(権限・ツール・なりすまし・RCE・供給網・記憶経由の漏出)。PB2(メモリ)/PB3(ツール実行)/PB4(認証)。
- **② 監査ログ/監視** → **T8(否認・追跡不能)が直接**。T4(リソース枯渇監視)・T10(HITL過負荷)。PB1 Step3(改ざん耐性ログ・決定反転検知)/PB6(エージェント間通信ログ)。
- **③ 著作権/法的適合性** → T1/T5(汚染→生成物の不正確・不適法)・T7(欺瞞)・T15(人間操作による不適切生成物の承認)。
