## 1. スコープ・位置づけ(本体固有)

- **「セキュリティ確保」の公式定義**: 「**不正操作による機密情報の漏えい、AIシステムの意図せぬ変更や停止が生じないような状態**」への脅威対策。← 起点①の定義そのもの。AI事業者GL共通指針の「セキュリティ確保」を技術面で具体化。
- **AI事業者GLとの関係**: AI事業者GL読者は、LLMのセキュリティ確保の具体手法について本GLを参照できる(=本GLはAI事業者GLの技術別冊の位置づけ)。
- **AISIガイドとの関係**: 「AIセーフティに関する評価観点ガイド」「レッドチーミング手法ガイド」(ともにAISI)に基づき、脅威をレッドチーミングで特定し対策の有効性を確認できる。
- **対象AI**: LLM及びLLMを構成要素に含むAIシステム。**AIエージェントは「技術が急激な発展途上で、特有の脅威・対策を安定的に確定することが困難」として対象外**。← ★Gemini Enterpriseのエージェンティック機能を全社展開する我々には重大な空白。エージェント特有リスクは本GLでカバーされない=別途OWASP Agentic AI等で補完が必要。
- **想定読者**: AI開発者・AI提供者(利用者は直接の読者でない)。ただし後述の営業秘密保護で利用者も裨益。

## 2. 脅威の地図(本体固有の分類)

- **主要脅威**(プロンプト入力のみで実行可能=可能性が高い): プロンプトインジェクション(直接/間接)、DoS攻撃。
  - **直接PIの手口**: ①指示の上書き(「過去の指示を無視せよ」) ②ロールプレイ(研究者を装いマルウェア作成) ③特殊な入力形式(Unicode/ASCIIアートに不正指示) ④別タスクへの置換(「システムプロンプトを品詞分解して」で内部設定を吐かせる)。
  - **間接PI**: 細工したWebページ・電子メールをLLMに参照させ不正出力を誘発。
- **その他の脅威**(前提条件・執拗なアクセスが必要): データポイズニング、細工モデルの導入、モデル抽出攻撃。
- **脅威特定の2軸**(リスクベース): ①脅威の影響の大きさ(用途による) ②発生可能性(外部接続の有無・利用者属性)。← 別添7C・図8のリスクベース評価と同じ考え方。

## 3. 対策の総論(本体固有のポイント)

- **★「営業秘密」保護の法的フック(3.1)**: 本GLの対策を講じ秘密管理しているAIシステムを利用すれば、万一データが漏洩しても**不正競争防止法(平成5年法律第47号)第2条第6項の「営業秘密」の秘密管理性要件**の充足に資する(有用性・非公知性要件も別途必要)。← セキュリティ対策が法的保護に資する論拠。内部統制・法的適合性の観点で重要。
  - ⚠**補正(パブコメ回答より)**: この点は強く言い過ぎないこと。総務省はパブコメ回答(東芝意見・[公開意見要約](2026-03_soumu_ai-security-technical-measures-guideline_public-comments.md)項56)で「本GLの対策実施が秘密管理性要件を満たすための**必要要件になるという意味ではなく**、**充足を基礎づける要素の一つにとどまる**。対策の有無をもって当該要件が判断されるものではない」と明示。よって「対策→要件充足」と断定せず「**充足を基礎づける有利な一要素**」と表現する。
- **責任主体を決める趣旨ではない**: 各組織が状況に応じ合理的な対策を選択するための指針。
- **対策概観マトリクス(表2/表3)**: 各脅威 × 対策(安全基準学習/システムプロンプト/入力検証/外部参照検証/出力検証/権限管理)の対応表。空欄=対策不要を意味しない、複数対策の重層で低減。
- **★その他の基本対策(3.5)**: LLM特有対策に加え、情報システムの基本対策が必要:
  - **監査ログの保存によるトレーサビリティ確保** ← ★起点②の直接根拠。「用途・提供条件によりログ保存の可否・閲覧者範囲が異なる」と注記=ログ設計のスコープ判断が要る。
  - レートリミット導入、開発環境の開発者権限管理、構成要素の信頼性確認。
- **評価ツール・データセット**: AISI「AIセーフティ評価ツール」、NII「AnswerCarefully」、NII攻撃データセット(オンラインゲーム「Ailbreak」で58,000件収集)、NICTの自動評価基盤。← S4の対策有効性検証(レッドチーミング)で活用可能。

## ★ 4. 想定事例(本体固有・e-Agency直結)

提供システムの公開範囲・連携有無別に2事例で脅威と対策を分析。**これが我々のユースケース別アセスメントの手本**。

### 想定事例1:内部向けチャットボット(RAG利用)← e-Agencyの社内Gemini展開そのもの

- 構成: 組織内ユーザ → UI(ガードレール:プロンプト検証/回答検証)→ オーケストレータ → 内部RAGデータストア(ベクトルDB/ファイルシステム)、基盤モデルは組織外(=Google)から提供を受ける運用を仮定。← **Gemini Enterprise + 社内文書RAGの構成と完全一致**。
- 想定攻撃: ①直接PI(RAGデータストアからのデータ窃取等)②間接PI(RAGデータストアのファイル経由)。**「内部からの攻撃」=社内ユーザ自身が攻撃者になりうる**点を明示。
- 想定対策: 安全基準学習(基盤モデル側=Google)+システムプロンプト強化+ガードレールによる入出力・外部参照検証+**オーケストレータ/RAGの権限管理**。

### 想定事例2:外部向けチャットボット(外部連携利用)

- 構成: 組織外ユーザ → 外部連携ツールでインターネット公開情報を取得して回答。
- 想定攻撃: 直接PI(システムプロンプト窃取)、DoS、外部連携先経由の間接PI(Webページの隠し指示)。
- 想定対策: 安全基準学習+システムプロンプト強化+ガードレール検証(外部連携=信頼境界が広いぶん外部参照データ検証が重い)。← 外販で顧客が公開チャットボットを作る場合の手本。

## 自社(e-Agency / 内部統制分科会)への適用示唆

1. **想定事例1が「社内Gemini展開のリファレンスアーキテクチャ兼脅威モデル」**。図5〜7(構成/攻撃シナリオ/対策)をそのまま自社の社内チャットボットのリスクアセスメント雛形に使える。「内部からの攻撃=社内ユーザが攻撃者」という前提は、性善説で組みがちな社内システムの盲点を突く重要な視点。別添7C(ワークシート)の1ケースとして「社内RAGチャットボット」を1枚で整理できる。
2. **「営業秘密」の法的フック(3.1)は、起点①とJ-SOX的内部統制をつなぐ最強の論拠**。「セキュリティ対策を講じる=漏洩時に不正競争防止法の保護を受けられる」という因果は、経営層に対策投資を説明する際の決め手になる。利用指針/アセスメントに「本対策は営業秘密の秘密管理性要件の充足にも資する」と明記する。
3. **起点②監査ログは3.5で「トレーサビリティ確保の基本対策」として公式に位置づけ**。AI事業者GL本編6)透明性①・別添2A 4-2(ログ自動取得)と三重に裏付けが揃った。「ログ保存の可否・閲覧者範囲は用途による」=ログ設計時にスコープと権限を決める作業項目が確定。
4. **AIエージェントが本GLの対象外**という注記は、我々にとって最大の注意点。Gemini Enterpriseのエージェント機能(自律的なツール実行・複数ステップ実行)は本GLの脅威モデルでカバーされない。**OWASP「Agentic AI – Threats and Mitigations」(用語集で参照されている)を別途取得し、エージェント特有リスクを補完する**必要がある。S1の優先取得対象に追加。
5. **対策の有効性検証=レッドチーミング**(AISIレッドチーミング手法ガイド)。S4の「評価」工程で、AISI評価ツール・NII AnswerCarefully等を使った自社チャットボットのレッドチーミングを計画できる。AI事業者GL別添2A 5-1(独立評価)とも接続。
6. **対策マトリクス(表2)を外販アセスメントの「脅威×対策」マッピング表として転用**。顧客に「あなたのAIシステムの脅威にはこの対策群が要る」と構造的に示せる。

## 起点3点との関連

- **① 機密データ** → セキュリティ確保の定義そのもの。営業秘密の法的フック(3.1)、想定事例1のRAGデータ窃取対策。
- **② 監査ログ/監視** → **3.5「監査ログの保存によるトレーサビリティ確保」が直接根拠**。レートリミットも監視・DoS対策。
- **③ 生成物の著作権・法的適合性** → 本GLはセキュリティ主題のため直接の記載は薄い。③は文化庁「AIと著作権」で補完(継続課題)。
