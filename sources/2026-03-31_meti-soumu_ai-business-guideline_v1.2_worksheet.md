## ★ ワークシートの列設計(= アセスメントシートの確定フォーマット)

| 列  | 見出し                                                             | 内容                                                     |
| --- | ------------------------------------------------------------------ | -------------------------------------------------------- |
| B   | (大分類見出し)                                                     | 例「AIシステム・サービス利用時に確認する事項」           |
| C   | 分類                                                               | 指針番号(例 2)②、5)① )                                   |
| D   | (項目名)                                                           | 例「適正利用」「セキュリティ対策」                       |
| E   | 検討にあたって重要な事項                                           | ガイドラインの設問(a.b.c.…)= 別添7チェックリストの詳細版 |
| F   | 対応箇所                                                           | 本編の該当部(例「5部 / 5)セキュリティ確保」)             |
| G   | **各自の事業において検討対象とする事項(該当しない場合はその理由)** | ★自社で記入。スコープ判断+除外理由                       |
| H   | 他の主体との関係についての事項                                     | 連携先(例「人材採用部門と連携の必要性あり」)             |
| I   | ✓                                                                  | チェック欄                                               |
| J   | **具体的なアプローチ**                                             | ★自社で記入。実際の対策                                  |
| K   | 最終検討日(見直し日)                                               | ★改訂管理。S4改訂ループに直結                            |

冒頭(R4-6)に**役割記入欄**: α 作成者 / β 実施状況の確認を行う者 / γ 責任者。記載例では「AI倫理委員会担当者」「人材採用担当者(AI倫理委員会がサポート)」が記入。← RACIの最小実装。

## ★ 記載例(AI利用者・採用AIケース)から学べる「書き方」

利用者シートは、A社グループの採用AI(エントリーシート合否支援)を題材に、各設問へどう答えるかを実演。当チームの記入見本になる:

- **2)② 適正利用**: 「a.採用AIの利用目的・範囲を理解し目的外利用しない / b.利用前の動作確認 / c.予測対象のエントリーシートが適切か確認 / d.担当者側で対応すべきリスクコントロールを認識」。連携=人材採用部門。
- **3)① 公平性**: 「ユーザー入力が推論性能に影響しないため**対象外**」← **G列で堂々と除外+理由**を書く運用が示されている(全項目対応不要の実例)。
- **4)① プライバシー**: 「前処理で個人名削除のため不適切入力に当たらず対象外」/「キャリアコンサル・申込者へデータ利用範囲と目的を説明し同意取得(通常の人事業務の内部統制として実施)」。
- **5)① セキュリティ**(★起点①): 「a.社内情報セキュリティルール遵守、疑義は開発部門へ報告 / b.**データ入力時に機密情報の不適切入力がないかダブルチェック**」← 起点①「機密データフィルタリング」の運用レベルの落とし所が具体的に書かれている。
- **7)② 説明 / 7)⑥ 文書化**(★起点②関連): 「申込者にAI利用を通知」「**問合せメールフォームを設置**」「提供文書を社内イントラで最新版閲覧可能に」「業務マニュアル策定」。
- **1)④ 多様性**: 「情報リテラシーが十分でない応募者向けに、採用AIとは別の人事チャネルを用意」。

## 自社(e-Agency / 内部統制分科会)への適用示唆

1. **これがS3の主要成果物の「器」**。`sources/` のExcelをコピーして `templates/` 配下に2系統を作る:
   - **社内版**: ガバナンスシート(利用指針策定の作業台)+ 利用者シート(全社の生成AI利用統制)。G/J/K列を自社の実態で埋める。
   - **外販テンプレ版**: 列とロジックはそのまま、記載例を顧客が穴埋めできる空欄に戻し、業種別サンプルを添える。
2. **G列の「該当しない場合の理由」運用が肝**。リスクベースで「やらないことを理由つきで明示」できる=過剰統制を避けつつ網羅性の説明責任を果たす。これはアセスメントの説得力の核。
3. **K列「見直し日」= 改訂ループ(S4)の実装ポイント**。各行に次回レビュー日を持たせれば、ワークシート自体が改訂トリガー管理台帳になる。監視ツールでこのK列を読んで期限アラートを出す設計が可能。
4. **役割欄(α/β/γ)= RACIの最小実装**。当チームをγ(責任者)、各分科会・現場をβ(確認者)に割り当てる雛形にできる。
5. **記載例5)①bの「ダブルチェック」= 起点①の人手側の落とし所**。技術的フィルタ(S4のDLP)と運用ルール(入力時ダブルチェック)の二段で機密データを守る、という現実的な設計指針が得られた。
6. **採用AIケースは社内の他分科会(営業・サポート等)の統制設計の手本**にもなる。各ユースケースをこのシートで1枚ずつ整理すれば、全社のリスクアセスメントが標準化できる。

## 起点3点との関連(ワークシート設問で具体化)

- **① 機密データ** → 利用者5)①(機密情報の不適切入力チェック・ダブルチェック)、4)①(個人情報・前処理での匿名化)。
- **② 監査ログ** → ガバナンスシート4(運用状況の記録)、利用者7)②⑥(通知・問合せ窓口・文書保管)。
- **③ 生成物の著作権** → 開発者/提供者シートの2)安全性・知財項目(本要約では利用者シート中心に確認、開発者シートに詳細あり)。
