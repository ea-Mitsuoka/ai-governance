## 1. エージェントAI = 生成AIと質的に異なる

- **AIエージェント定義**: 環境を感知し、意思決定し、リソースにアクセスし、ツール・APIを呼び出して**所定の目標を自律的に達成**するソフトウェア。LLM+機械学習を用いることが多い。
- 生成AI(プロンプトされて出力)との違い=**自律的にタスクを実行**(モデル・データ・システムを連結して目標を達成)。コンテンツ生成→自律実行への移行が、既存リスクを増幅し新リスクを生む。
- 採用は加速:調査対象の4割が既に本番導入、3年以内に4分の3が導入見込み。**全体採用率 ~43%→~76%(3年)**。

## 2. エージェントの6類型(自律度で分類)= リスクレベルの基礎

| 類型               | 説明                                                    | 自律度 |
| ------------------ | ------------------------------------------------------- | ------ |
| Knowledge/research | 情報提供・分析(市場調査要約等)                          | 低     |
| Task               | 単純・離散的アクション(会議設定・通知)                  | ↓      |
| Workflow           | 複数ステップの業務プロセス実行(オンボーディング等)      | ↓      |
| Goal/planning      | 計画立案・実行経路決定(人員計画・投資戦略)              | ↓      |
| **Orchestrator**   | **複数エージェント/システムを統括**(リリース管理・移行) | 高     |
| Personable         | 人間らしい対話・意思決定(顧客対応・研修)                | 高     |

→ ★**統制の第一歩=「エージェント類型とリスクレベルの分類体系(taxonomy)を組織で確立する」**。研究系は低リスク、オーケストレータは高リスク(複数システムへ作用)。これは別添7C(ワークシート)で各ユースケースを1枚ずつ整理する我々のやり方と相性がよい。

## ★ 3. エージェントAIの新リスク類型(Exhibit 5)= リスク台帳に追加すべき5項目

| 新リスク(伝統的アンカー)                          | 説明                                                        | 例                                                                        |
| ------------------------------------------------- | ----------------------------------------------------------- | ------------------------------------------------------------------------- |
| **Cross-agent task escalation**(権限昇格)         | エージェントが意図範囲を超えてタスク/権限を委譲             | 不正スケジューリングAgentが医師AgentになりすましてカルテをリリースさせるI |
| **Untraceable data leakage**(データ持ち出し)      | エージェント/ツール間の機微データ流通がログ・監査なしに発生 | サポートAgentが外部「不正検知」Agentへ口座情報を無記録で共有              |
| **Synthetic identity risk**(なりすまし)           | 偽造・複製されたAgent IDで信頼/説明責任を回避               | 攻撃者がクレーム処理AgentのIDを偽造し正規扱いさせる                       |
| **Chained vulnerabilities**(サプライチェーン欠陥) | 複数Agent/ツールの小さな弱点が連鎖し系統的悪用に            | DevOps OrchestratorがデプロイポリシーをⅮ誤読しadmin権限でコンテナ起動     |
| **Propagation of corrupted data**(データ汚染)     | 汚染・ポイズニングされたデータがAgentワークフローを自動伝播 | 誤ラベルのAgentがⅮ下流の効能・規制報告Agentを汚染                         |

- **シングルエージェント vs マルチエージェント**(Exhibit 4): 単一Agentならリスクは境界内に封じ込まる。**マルチAgent(協調)では相互作用が新たなリスク経路を生み、障害・汚染データが連鎖伝播**する。「目に見えないエージェント間データフロー」が検出・封じ込めを困難にする。
- CISO調査:\*\*3分の2が「エージェントAI」をトップ3のサイバーリスクに、3分の1超が"最大の懸念"(ランサムウェア・サプライチェーンより上位)\*\*に挙げた。

## ★ 4. 拡張すべき統制の4領域(=S4の設計要件)

伝統的統制(IAM・RBAC・DLP)は静的ユーザ・固定アクセス前提で、自律システムには部分的保護にとどまる。4領域を追加:

1. **Identity & access controls**: エージェントの**作成・認証・廃棄**を管理。**エージェントレジストリ**(所有者・権限・ライフサイクルの権威記録)、**動的Just-In-Time(JIT)アクセス権限**(タスク・権限・リスクに応じ自動調整)。
2. **Visibility & monitoring controls**(★起点②): エージェント活動を**追跡・記録**し追跡可能・監査可能に。**継続的アクティビティログ、エージェント異常検知ダッシュボード**。
3. **Containment & recovery controls**: 侵害エージェントを隔離・停止。**キルスイッチ、安全状態への自動ロールバック**。
4. **Data assurance controls**(★起点①③): データの出所・アクセス可否・使われ方を検証。**ソース検証ツール、機微情報へのアクセス承認**。

→ ⚠ギャップ警告:採用は3年で76%へ向かうのに、**リスクレジストリと動的認可を本番展開済みの組織は1割未満**。一方60%超のCISOが「これらは重要」と認識。**自律性が封じ込め策を追い越す**リスク。

## 5. アイデンティティ=制御の中心(本書の主張、ベンダーポジション)

- アイデンティティが「認証」から「オーケストレーション」へ進化。エージェントのライフサイクル全体(作成・権限付与・封じ込め)を司る制御プレーンに。
- エージェントのアイデンティティ=人間+マシンの特性に加え、**文脈的自律性・委譲された権限・非決定論的振る舞い**という新次元 → 新たなガバナンス機構が要る。
- ガバナンス・データが「共同制御プレーン」に(EU AI Act・DORA等の規制が牽引、予算の約1/3を吸収)。境界(ペリメータ)型統制は後退。
- ⚠これは「アイデンティティ製品(Idira)が解」というベンダー誘導。**論旨は妥当だが製品選定は中立評価が必要**。

## 6. ガバナンスの3類型(体制設計の選択肢)

- **Centralized**(Chief AI Officer/AIリスク委員会): 単一の責任集中、一貫した基準・規制整合。AIが戦略・収益の中核/取締役会可視性が必要な場合に有効。レビュー層追加で実行が遅くなりうる。
- **Functional**(CRO/CLO/CISO主導): 既存機能に責任を埋め込む。成熟したリスク管理枠組みを持つ組織向け。横断的ギャップが残りうる。
- **Federated**(事業部・地域主導): 柔軟・迅速。多角化/デジタルファースト組織向け。中央調整がないと基準が不揃いに。
- → e-Agencyは小〜中規模なので、当面\*\*Functional(当チーム=内部統制/BSM室に責任を埋め込む)\*\*が現実的。将来エージェント利用が中核化すればCentralized(AI倫理委員会)へ寄せる、という移行設計ができる。

## 自社(e-Agency / 内部統制分科会)への適用示唆

1. **総務省GLの空白(AIエージェント対象外)を、本書のリスク類型・統制4領域で実務的に埋められる**。ただしベンダー資料なので、中立標準(OWASP Top 10 for Agentic Applications、NIST SP 800-53 AI control overlays、本書が参照)での裏取りは継続課題。
2. **リスク台帳に「エージェント特有の5リスク」を追加**: cross-agent task escalation / untraceable data leakage / synthetic identity / chained vulnerabilities / corrupted data propagation。総務省GLのプロンプトインジェクション・DoS等に、この5つを重ねる。
3. **起点3点とエージェントリスクの対応が取れた**:
   - 起点①機密データ → **untraceable data leakage**(エージェント間の無記録データ流通)+ data assurance controls。総務省GLのRAGアクセス制御に「エージェント間データフローの可視化」を追加。
   - 起点②監査ログ → **visibility & monitoring controls**(継続的アクティビティログ・異常検知ダッシュボード)。「エージェントの行動ログ」は単一LLMの入出力ログより取得対象が広い=監視設計の拡張要件。
   - 起点③著作権 → corrupted data propagation(汚染データの伝播)経由で、生成物の適法性管理にエージェント連鎖の観点が要る。
4. **S4のガードレール設計に4つの新統制を要件追加**: ①エージェントレジストリ(誰が・何の権限で・ライフサイクル) ②動的JITアクセス ③キルスイッチ・自動ロールバック ④ソース検証。総務省GLの「最小権限・オーケストレータ権限管理」を、エージェント単位のID統制へ拡張。
5. **「マルチエージェントは封じ込めが効かない」が最重要の設計原則**。Gemini Enterpriseでエージェント同士を連携させるほどリスクが連鎖伝播する。当面は\*\*単一エージェント境界を保つ(高自律・オーケストレータ型の多段連携は慎重に)\*\*設計判断が、リスクベースで妥当。
6. **体制はFunctionalガバナンスから**(当チームに責任埋め込み)。エージェント利用が中核化したらCentralized(AI倫理委員会)へ移行する設計。AI事業者GL別添2A 1-1のAI倫理委員会論と接続。
7. **「自律性が封じ込めを追い越す」警告は、導入速度のガバナンス論拠**。全社展開のペースを、レジストリ・監視・キルスイッチの整備状況に合わせて律速する(統制が追いつかないまま自律エージェントを広げない)。

## 起点3点との関連

- **① 機密データ** → untraceable data leakage、data assurance controls、エージェント間データフロー可視化。
- **② 監査ログ/監視** → visibility & monitoring controls(アクティビティログ・異常検知)。エージェント行動ログへの拡張。
- **③ 著作権** → corrupted data propagation経由で生成物適法性のエージェント連鎖管理。
